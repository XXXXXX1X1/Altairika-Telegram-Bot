## 2. Компоненты

### 2.1 Бот-сервис (основной процесс)

**Технологии:** Python 3.11+, [aiogram 3.x](https://docs.aiogram.dev/)

Aiogram выбран как зрелый async-фреймворк для Telegram с поддержкой:
- FSM (конечные автоматы для форм),
- middleware,
- удобного роутинга.

**Режим приёма обновлений:** polling.
- Текущая реализация запускает `Dispatcher.start_polling(...)`.
- Для локальной разработки и простого VPS этого достаточно.
- Переход на webhook возможен позже, но в коде сейчас не реализован.

**Структура модулей:**

```
bot/
├── handlers/          # Обработчики событий
│   ├── start.py
```

### 2.2 База данных

**PostgreSQL** (см. модель данных в документе 04).

**ORM:** SQLAlchemy (async) + Alembic для миграций.

> [Гипотеза] На очень раннем прототипе — SQLite + синхронный SQLAlchemy. Переход на async PostgreSQL при первом деплое.

---

### 2.3 Кэш

**На старте:** in-memory кэш в процессе (functools.lru_cache или простой dict).

**При росте нагрузки [Улучшение]:** Redis.

Что кэшируется:
- Список категорий каталога (редко меняется).
- Список карточек по категориям.
- Контент разделов франшизы.
- Данные для сравнения конкурентов.

TTL кэша: 1–6 часов (настраивается).

---

### 2.4 FaqService

**Назначение:** навигация по FAQ и сохранение вопросов пользователя оператору.

**Логика:**
- Список тем и вопросов загружается из БД, кэшируется.
- Свободный текстовый ввод вне формы → предлагает FAQ или «написать вопрос оператору».
- Вопрос оператору сохраняется в `UserQuestion`, оператор получает уведомление.
- Никакого разбора намерений, никакого keyword-matching — только структурированный FAQ.

### 2.5 Парсер каталога

**Отдельный процесс / cron-задача.** Не часть бот-сервиса.

```
Scheduler (APScheduler или cron)
        │
        ▼
HTTP запрос к altairika.ru/catalog_full
        │
        ▼
Извлечение recid/storepart из HTML страницы
        │
        ▼
Запросы к Tilda API /api/getproductslist/
        │
        ▼
Нормализация данных
        │
        ▼
Сравнение с текущим состоянием БД
        │
        ▼
Обновление / добавление / деактивация записей
        │
        ▼
Инвалидация кэша бота
```

> [Решение] Текущий парсер не использует Playwright и не разбирает DOM-карточки сайта. Он читает страницу `catalog_full`, находит блоки каталога Tilda Store и затем забирает товары через Tilda API.

> [Решение] Если на странице найдено несколько блоков каталога, парсер выбирает канонический блок с максимальным `total`. На текущем сайте это каталог на 166 фильмов.

> [Решение] Парсер запускается 1 раз в сутки ночью через APScheduler. Дополнительно есть ручной запуск скриптом `python scripts/run_parser.py`.

> [Решение] CSV-импорт остаётся резервным способом наполнения, если структура сайта снова изменится.

### 2.6 Уведомления оператору

При поступлении новой заявки (`Lead`) — бот отправляет уведомление в отдельный Telegram-чат / канал администратора.

```
Новая заявка создана
        │
        ▼
Telegram сообщение администратору:
───────────────────────────────
🆕 Новая заявка [тип]

Имя: [Имя]
Телефон: [Номер]
Позиция: [Название или —]
Время: [Пожелание или —]
───────────────────────────────
```

> [Решение] Простейший вариант уведомления. CRM-интеграция — [Улучшение] второй фазы.

## 3. Деплой

> [Решение] Минимальная инфраструктура для старта:

| Компонент | Вариант |
|-----------|---------|
| Хостинг бота | VPS (минимум 1 CPU, 1 GB RAM) |
| БД | PostgreSQL на том же VPS или managed (Yandex Cloud / Timeweb) |
| Контейнеризация | Docker + docker-compose |
| CI/CD | GitHub Actions (опционально на старте) |
| Домен и TLS | Не требуется для текущего polling-режима |
| Логирование | Python logging → файл + опционально Sentry |

> [Гипотеза] Managed PostgreSQL (например, Yandex Managed PostgreSQL) удобнее на продакшне, но дороже. На старте — PostgreSQL в Docker-контейнере достаточно.

---

## 4. Безопасность

- Токен бота и ключи API хранятся только в переменных окружения (`.env` файл, не в репозитории).
- SQL-запросы только через ORM (защита от инъекций).
- Валидация номера телефона перед сохранением.
- Rate limiting на свободные текстовые запросы (защита от злоупотребления LLM).

> [Улучшение] Шифрование персональных данных в БД — второй фазы приоритет.

---

## 5. Масштабируемость

Текущая архитектура рассчитана на низкую-среднюю нагрузку (сотни пользователей в сутки).

При росте нагрузки (без переписывания):
- Добавить Redis для кэша и FSM-состояний.
- Масштабировать бот горизонтально (несколько инстансов с общим Redis).
- Перейти на managed PostgreSQL с репликой для чтения.

> [Гипотеза] Для данного проекта вертикальное масштабирование (более мощный VPS) достаточно на 12–18 месяцев после запуска.
