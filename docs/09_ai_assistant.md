# 09. AI-ассистент

## Цель

Добавить в бота AI-ассистента для свободных текстовых запросов без разрушения текущей навигации по кнопкам.

Ассистент должен:

- консультировать по вопросам о компании;
- помогать подбирать фильмы по возрасту, теме, длительности и аудитории;
- отвечать по франшизе;
- объяснять отличия от конкурентов;
- переводить пользователя в заявку, если он готов оставить контакт;
- не выдумывать данные и не отвечать вне предметной области.

Ключевой принцип: AI не должен быть единственным источником данных. Модель выступает как слой понимания запроса и генерации ответа, а факты берутся из существующей базы данных, API и базы знаний.

---

## Что меняется в продукте

### Точка входа

Основная точка входа — `bot/handlers/freetext.py`.

Сейчас `freetext_router` отвечает заглушкой. После внедрения AI этот роутер:

1. принимает свободный текст;
2. определяет сценарий запроса;
3. собирает только релевантный контекст;
4. вызывает модель;
5. возвращает ответ и подходящие кнопки.

### Что не меняется

- FSM-формы заявок и вопросов оператору работают как раньше;
- callback-обработчики каталога, FAQ, франшизы и сравнения не ломаются;
- команды `/start`, `/admin` и служебные сценарии сохраняются;
- AI не подменяет собой каталог и админку.

---

## Продуктовые сценарии

AI должен поддерживать не один общий сценарий, а несколько разных.

### 1. Подбор фильма

Примеры:

- «Подберите фильм для начальной школы»
- «Что есть про космос для 3 класса?»
- «Нужен короткий фильм до 15 минут»

Ожидаемое поведение:

- определить параметры запроса;
- найти релевантные фильмы в `catalog_items`;
- вернуть 3-5 подходящих вариантов;
- кратко объяснить, почему они подходят;
- если данных не хватает, задать один уточняющий вопрос.

### 2. Общая консультация о компании

Примеры:

- «Что такое Альтаирика?»
- «Как проходит сеанс?»
- «Где можно провести показ?»

Ожидаемое поведение:

- использовать базу знаний о компании и FAQ;
- отвечать кратко, без галлюцинаций;
- в конце при необходимости предлагать перейти в каталог или оставить заявку.

### 3. Франшиза

Примеры:

- «Сколько стоит франшиза?»
- «Что входит в пакет?»
- «Для маленького города подойдет?»
- «Какая окупаемость?»

Ожидаемое поведение:

- брать факты из `franchise_content` и базы знаний;
- отвечать только по имеющимся данным;
- если вопрос ведет к продаже, завершать CTA на заявку.

### 4. Сравнение с конкурентами

Примеры:

- «Чем вы лучше VR Concept?»
- «Сравните с Vizerra»
- «Какие преимущества перед конкурентами?»

Ожидаемое поведение:

- брать данные из `competitors`, `comparison_parameters`, `comparison_values`;
- строить ответ на основе существующей таблицы сравнения;
- не придумывать сравнений, которых нет в данных.

### 5. FAQ и типовые вопросы

Примеры:

- «Безопасен ли VR для детей?»
- «Сколько человек может смотреть одновременно?»
- «Что нужно подготовить для показа?»

Ожидаемое поведение:

- сначала проверять `faq_items`;
- если FAQ не нашелся, использовать knowledge base;
- если ответа нет нигде, честно сказать об этом и предложить задать вопрос оператору.

### 6. Намерение оставить заявку

Примеры:

- «Хочу записаться»
- «Свяжитесь со мной»
- «Хочу обсудить франшизу»

Ожидаемое поведение:

- не пытаться долго консультировать;
- перевести пользователя в существующую FSM-форму;
- сохранить текущий сценарий lead generation.

### 7. Смешанные вопросы

Примеры:

- «Подберите фильм для 3 класса и расскажите, как проходит показ»
- «Что есть про космос и сколько это стоит?»

Ожидаемое поведение:

- уметь собрать ответ из нескольких источников;
- сначала решить основную задачу пользователя;
- затем дать короткий дополнительный контекст;
- не перегружать ответ.

### 8. Off-topic

Примеры:

- «Как приготовить борщ?»
- «Какая погода завтра?»

Ожидаемое поведение:

- вежливо отказать;
- вернуть пользователя к тематике бота.

---

## Архитектурный принцип

На старте нельзя отправлять модели всю базу данных целиком на каждый вопрос.

Это плохой вариант, потому что он:

- дороже по токенам;
- хуже масштабируется;
- сильнее подвержен галлюцинациям;
- усложняет контроль над источником ответа.

Вместо этого используется схема:

`intent router -> source selection -> context builder -> LLM answer`

Где:

- `intent router` понимает тип запроса;
- `source selection` выбирает нужный источник данных;
- `context builder` собирает небольшой релевантный контекст;
- `LLM answer` формирует финальный ответ пользователю.

---

## Источники данных

AI не должен работать только на одном “большом документе”. Источники нужно разделять по типам.

### 1. Структурированные данные из БД

Используются там, где нужен точный отбор, фильтрация и сравнение.

| Задача | Источник |
|--------|----------|
| Подбор фильмов | `catalog_items`, `categories` |
| FAQ | `faq_topics`, `faq_items` |
| Франшиза | `franchise_content` |
| Сравнение с конкурентами | `competitors`, `comparison_parameters`, `comparison_values` |
| Переход в форму заявки | существующие FSM и `leads` |

### 2. Общая база знаний

Используется для текстовой консультации и контекстных ответов.

Рекомендуемый формат первой версии:

- один документ `docs/company_knowledge.md`

В нем хранить:

- описание компании;
- как проходит сеанс;
- ценностное предложение;
- особенности работы со школами, родителями, мероприятиями;
- преимущества бренда;
- правила ответа модели;
- оговорки, что нельзя обещать без подтверждения;
- типовые возражения и ответы;
- конверсионные формулировки.

### 3. Внешний API

Если у проекта уже есть API для каталога или других бизнес-данных, он должен считаться источником истины.

Рекомендация:

- API хранит и отдает актуальные сущности;
- PostgreSQL кеширует или материализует нужные данные для бота;
- AI работает поверх актуального слоя данных, а не поверх произвольного текста.

---

## Почему каталог нельзя решать только базой знаний

Подбор фильма — это не просто “найти похожий текст”.

Запросы пользователя часто содержат параметры:

- возраст;
- класс;
- тема;
- длительность;
- формат;
- тип мероприятия.

Поэтому подбор фильма должен работать так:

1. Извлечь параметры из пользовательской фразы.
2. Найти кандидатов в `catalog_items`.
3. Передать в модель только найденные релевантные фильмы.
4. Попросить модель сформулировать понятную рекомендацию.

Иначе модель:

- будет путаться в большом каталоге;
- начнет рекомендовать несуществующие фильмы;
- будет плохо объяснять, почему выбор подходит.

## Intent Router

### Цель

Перед ответом система должна понять, какой именно сценарий запустил пользователь.

### Минимальный набор intent'ов

| Intent | Когда используется |
|--------|--------------------|
| `movie_selection` | Подобрать фильм, рекомендации из каталога |
| `movie_details` | Вопрос о конкретном фильме |
| `company_info` | Вопросы о компании и формате показов |
| `franchise_info` | Вопросы о франшизе |
| `competitor_compare` | Сравнение с конкурентами |
| `faq_answer` | Типовой вопрос, похожий на FAQ |
| `lead_booking` | Намерение оставить заявку на показ |
| `lead_franchise` | Намерение обсудить франшизу |
| `operator_handoff` | Нужен человек, а не AI |
| `off_topic` | Запрос не относится к предметной области |

### Как определять intent

На MVP достаточно одного из двух вариантов:

1. Правила + ключевые слова
2. Малый LLM-классификатор

Практически лучше использовать гибрид:

- сначала простые правила на явные случаи;
- потом LLM-классификатор для неоднозначных запросов.

## Память и история диалога

### Нужна ли история

Да, но не в формате “отправлять модели весь диалог целиком”.

Для этого проекта полезна краткосрочная память:

- пользователь спрашивает в несколько шагов;
- бот постепенно уточняет параметры;
- потом выдает релевантный результат.

### Где память реально нужна

#### Подбор фильма

Пример:

- «Подберите фильм для начальной школы»
- «Про космос»
- «И чтобы не длинный»

Без памяти это три независимых запроса. С памятью это один сценарий подбора.

#### Франшиза

Пример:

- «Расскажите про франшизу»
- «А для маленького города подойдет?»
- «А какая окупаемость?»

Здесь нужно помнить, что активен сценарий `franchise_info`.

### Как хранить память правильно

На старте хранить не всю историю сообщений, а сжатое состояние диалога:

- активный intent;
- извлеченные параметры;
- краткую сводку последнего шага;
- время последней активности.

Это дешевле и надежнее, чем полная chat history.

### Что хранить для подбора фильма

Пример `session_state`:

```json
{
  "intent": "movie_selection",
  "audience": "начальная школа",
  "grade": "3 класс",
  "theme": "космос",
  "duration": "d15",
  "age_rating": "7+",
  "last_recommended_ids": [12, 47, 88]
}
```

### Где хранить

На MVP допустимо:

- в отдельной таблице PostgreSQL;
- или в FSM storage, если память нужна только на короткую сессию.

Для production-пути лучше отдельная таблица, так как:

- память не пропадет при перезапуске;
- можно анализировать сценарии;
- не нужно привязываться к MemoryStorage.

### Вывод по стоимости

Память сама по себе не дорогая. Дорого становится только если на каждый запрос передавать модели весь накопленный чат.

Правильный подход:

- хранить состояние диалога отдельно;
- в модель отправлять только короткую выжимку и текущие релевантные данные.

## Логика обработки запроса

### Общий pipeline

```text
Пользователь отправляет текст
        ↓
freetext_router принимает сообщение
        ↓
Проверка: нет ли активной FSM-формы
  Да → управление остается у формы
  Нет → продолжаем
        ↓
Загрузка session state пользователя
        ↓
Определение intent
        ↓
Выбор источников данных
        ↓
Извлечение параметров запроса
        ↓
Получение релевантных данных из БД / API / knowledge base
        ↓
Сборка короткого контекста
        ↓
Вызов LLM
        ↓
Ответ пользователю
        ↓
Обновление session state
```

### Ветвление по сценариям

#### `movie_selection`

1. Извлечь фильтры из запроса.
2. Объединить их с уже сохраненными параметрами из session state.
3. Найти релевантные фильмы в БД.
4. Если найдено достаточно вариантов:
   выдать рекомендации.
5. Если вариантов слишком много:
   задать один уточняющий вопрос.
6. Если ничего не найдено:
   предложить близкие альтернативы или помощь менеджера.

#### `company_info`

1. Проверить FAQ.
2. Если FAQ не дал ответа, использовать company knowledge.
3. Сформировать короткий ответ.

#### `franchise_info`

1. Загрузить нужные разделы `franchise_content`.
2. При необходимости дополнить knowledge document.
3. Вернуть ответ и CTA на форму франшизы.

#### `competitor_compare`

1. Найти упомянутого конкурента, если он назван.
2. Загрузить параметры сравнения.
3. Сформировать ответ из структурированных данных.

#### `lead_*`

1. Не продолжать консультацию слишком долго.
2. Перевести пользователя в существующую форму.

## Подбор фильма: детальная спецификация

Это ключевой сценарий, поэтому он требует отдельной логики.

### Что должен понимать ассистент

Из текста пользователя нужно уметь извлекать:

- возраст или школьный класс;
- тему;
- длительность;
- формат мероприятия;
- желаемую подачу: образовательный, зрелищный, спокойный, короткий;
- ограничения, если пользователь их указал.

### Примеры извлечения параметров

Запрос:

`Хочу подобрать фильм для детей начальной школы`

Извлечь:

- `intent = movie_selection`
- `audience = primary_school`
- `grade = null`
- `theme = null`
- `duration = null`

Запрос:

`Нужен фильм про космос для 3 класса до 15 минут`

Извлечь:

- `intent = movie_selection`
- `grade = 3`
- `theme = космос`
- `duration = d15`

### Алгоритм ответа

1. Если параметров достаточно:
   искать фильмы сразу.
2. Если параметров мало:
   сначала выдать 2-3 общих варианта и задать уточняющий вопрос.
3. Никогда не выдумывать названия фильмов.
4. Отвечать только на основе реальных записей из `catalog_items`.

### Формат ответа

Пример:

```text
Для начальной школы могу предложить несколько вариантов:

1. Название фильма
Коротко: почему подходит этой аудитории.

2. Название фильма
Коротко: в чем ценность и какая тема.

3. Название фильма
Коротко: кому лучше всего подойдет.

Если хотите, я могу уточнить подбор по теме: космос, природа, история или наука.
```

### Что передавать модели

Не весь каталог, а только:

- исходный запрос;
- session state;
- 3-10 лучших кандидатов из БД.

## База знаний компании

### MVP-формат

Первая версия может хранить знания в одном документе:

- `docs/company_knowledge.md`

### Что в нем должно быть

- позиционирование компании;
- как объяснять формат сеанса;
- какие формулировки использовать в ответах;
- что можно обещать клиенту;
- чего обещать нельзя;
- ответы на типовые вопросы, которых нет в FAQ;
- микро-скрипты для перехода к заявке;
- пояснения для школ, родителей, партнеров и франчайзи.

### Когда одного документа уже недостаточно

Если знаний становится много, нужно переходить на отдельную сущность knowledge base:

- `knowledge_articles`
- `knowledge_chunks`

Но это не обязательно для первого этапа.

## RAG и embeddings

### Нужны ли они на старте

Не обязательно.

Для вашего проекта на MVP обычно достаточно:

- PostgreSQL;
- структурированных таблиц;
- одного knowledge document;
- простого intent router.

### Когда RAG действительно нужен

Если:

- база знаний стала большой;
- появилось много длинных документов;
- нужен semantic search по неструктурированным материалам;
- один документ стал неудобным для поддержки.

### Если добавлять RAG позже

Лучший путь для этого проекта:

- оставить PostgreSQL;
- добавить `pgvector`;
- хранить embeddings рядом с остальными данными.

Рекомендуемые сущности:

- `kb_documents`
- `kb_chunks`

Но каталог фильмов и сравнение конкурентов даже после внедрения RAG лучше оставлять на structured retrieval, а не переводить полностью на vector search.

## Prompting

### Роль модели

Модель должна:

- понимать запрос пользователя;
- соблюдать ограничения домена;
- использовать только переданный контекст;
- отвечать кратко и дружелюбно;
- не выдумывать отсутствующие данные.

### Базовая инструкция

Системный промпт должен включать правила:

- Ты помощник компании Альтаирика.
- Отвечай только по темам компании, каталога, франшизы, FAQ и сравнения.
- Используй только факты из переданного контекста.
- Если данных недостаточно, скажи об этом прямо.
- Если вопрос не относится к компании, мягко верни пользователя к теме.
- Если пользователь явно хочет оставить заявку, предложи соответствующую форму.

### Важная поправка

В системный промпт нельзя подставлять всю базу целиком. В prompt попадает только отобранный контекст для конкретного запроса.

## Структура модулей

Рекомендуемая структура:

```text
bot/
├── handlers/
│   └── freetext.py
├── services/
│   ├── ai_router.py
│   ├── ai_context.py
│   ├── ai_catalog.py
│   ├── ai_memory.py
│   ├── ai_answer.py
│   └── ai_client.py
└── repositories/
    └── ai_sessions.py
```

### Ответственность модулей

#### `ai_router.py`

- определение intent;
- выбор сценария;
- решение, нужен ли AI-ответ, переход в FSM или handoff оператору.

#### `ai_context.py`

- сборка контекста из FAQ, франшизы, knowledge document и comparison-данных.

#### `ai_catalog.py`

- извлечение параметров подбора;
- поиск релевантных фильмов;
- ранжирование кандидатов.

#### `ai_memory.py`

- чтение и обновление session state;
- сборка краткой выжимки для модели.

#### `ai_answer.py`

- сборка сообщений для модели;
- общая логика генерации ответа.

#### `ai_client.py`

- вызов OpenRouter/OpenAI-совместимого API;
- таймауты;
- обработка ошибок.

## Session state

### Зачем отдельная таблица

Если AI должен помнить контекст между сообщениями, лучше не использовать только `MemoryStorage`.

Причины:

- состояние теряется при перезапуске;
- сложно анализировать сценарии;
- нет устойчивой памяти для production.

### Рекомендуемая таблица

Пример:

| Поле | Тип | Назначение |
|------|-----|-----------|
| `telegram_user_id` | bigint PK | Пользователь |
| `active_intent` | varchar(100) | Текущий сценарий |

## Обработка ошибок и fallback

Если AI или API не ответили:

- не ломать UX;
- вернуть пользователя в управляемый сценарий.

Рекомендуемый fallback:

- краткое сообщение о том, что ответить автоматически сейчас не получилось;
- кнопки на FAQ, каталог, форму вопроса оператору, главное меню.

Если не найдено данных:

- честно сказать, что точного ответа нет;
- предложить уточнить запрос или написать оператору.

## Стоимость

Основная стоимость зависит не от самого факта “есть память”, а от объема токенов, отправляемых в модель.

### Что делает решение дорогим

- отправка всей БД в prompt;
- отправка всего каталога на каждый запрос;
- отправка полной истории чата без сжатия.

### Что делает решение дешевым

- intent routing;
- выбор только нужного источника данных;
- краткая session memory;
- передача в модель только релевантных фильмов или фрагментов знаний.

Вывод: stateful-диалог в этом проекте допустим по стоимости, если хранить не полную историю, а краткое состояние пользователя.

---

## Рекомендуемый MVP

### Что входит

1. Свободный текст в `freetext_router`
2. Intent router
3. Подбор фильмов из БД
4. Ответы по компании из `company_knowledge.md`
5. Ответы по франшизе из `franchise_content`
6. Ответы по конкурентам из текущих таблиц
7. Краткосрочная память в виде session state
8. Fallback на оператора и FAQ

### Что не входит

- полноценная vector DB;
- длинная память диалога;
- персонализация по пользователю;
- автоматическое обучение на переписке;
- сложный multi-agent orchestration.

## Итоговое решение

Для этого проекта AI-ассистент должен быть построен не как “одна модель, которая читает всю БД”, а как управляемая система поверх существующей архитектуры.

Правильная схема:

- компания и общие ответы -> knowledge base;
- подбор фильмов -> структурированный поиск по каталогу;
- франшиза -> `franchise_content` + knowledge base;
- конкуренты -> таблицы сравнения;
- история -> краткое session state, а не полный чат;
- LLM -> только понимание запроса и формулирование ответа.

Такое решение:

- дешевле;
- точнее;
- проще поддерживается;
- лучше развивается от MVP к production.
