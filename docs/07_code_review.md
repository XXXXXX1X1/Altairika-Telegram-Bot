# Code Review Report

> Дата: 2026-04-08

## Резюме

Кодовая база хорошо структурирована, соответствует заявленной архитектуре и уровню проекта. Большинство найденных проблем относятся к категории «средняя/низкая» — критических уязвимостей нет. Основные риски: N+1-паттерн в каталоге (перезапрос всех строк при каждом листании), непойманная NameError в production-пути FAQ, и жёстко захардкоженные учётные данные БД в docker-compose.

## Проблемы по модулям

### bot/handlers/faq.py

- **[критичность: высокая] Корректность: NameError в production-пути.**
  `faq.py:166` — переменная `question` не определена в этой области видимости. `save_user_question` возвращает объект, но он нигде не сохраняется (`await save_user_question(...)` без присваивания), а строкой ниже используется `question.id`. Это упадёт с `NameError` при первой же отправке вопроса пользователем.
  ```python
  # Текущий код (строка 152–167):
  await save_user_question(session, ...)
  await state.clear()
  # ...
  callback_data=AdminQCb(..., q_id=question.id).pack()  # NameError: question
  ```
  **Исправление:** `question = await save_user_question(session, ...)` на строке 152.

### bot/repositories/catalog.py

- **[критичность: средняя] Производительность: N+1 / полная загрузка таблицы для каждого запроса листания.**
  Функции `count_active_items` и `get_items_page` оба вызывают `get_filtered_active_items`, которая делает `SELECT * FROM catalog_items WHERE is_active`. Это означает, что при каждом нажатии «следующая страница» из БД вычитывается **весь** каталог, после чего фильтрация по `age_rating`, `duration` и JSON-тегам происходит в Python. При каталоге в несколько сотен позиций и одновременных пользователях это создаёт ненужную нагрузку.
  Кроме того, `count_active_items` вызывает `get_filtered_active_items` и затем берёт только `len()` — т.е. одна из двух идентичных выборок всегда выбрасывается.
  **Исправление (приемлемое для текущего масштаба):** объединить `count` и `get_page` в один вызов, чтобы таблица читалась один раз. Долгосрочно — перенести фильтрацию по `age_rating` и `duration` на уровень SQL, сохранить JSON-фильтрацию в Python.

- **[критичность: низкая] Производительность: многократный вызов `get_filtered_active_items` за один request.**
  В `_render_list` (catalog.py handler) вызываются последовательно `count_active_items`, `get_items_page`, `get_active_categories` и `get_available_genre_filters`. Первые два по отдельности читают всю таблицу. Итого — минимум 3 полных `SELECT` за одно нажатие кнопки.

- **[критичность: низкая] Корректность: `theme_key` использует MD5 без учёта регистра/нормализации.**
  `theme_key(value)` вычисляется от сырой строки темы (`md5(value.encode())`). Если та же тема сохранена с разным регистром или пробелами в разных карточках, хеши не совпадут, и фильтр «Похожие» перестанет работать корректно.
  **Исправление:** нормализовать строку перед хешированием: `value.strip().lower()`.

### bot/handlers/start.py

- **[критичность: средняя] Производительность: `/sync` создаёт новый `create_async_engine` при каждом вызове, не закрывая старый пул.**
  Строки 121–123: внутри обработчика команды создаётся новый engine и session_factory. `await engine.dispose()` вызывается корректно, но только при нормальном завершении. Если `sync_catalog` бросит исключение — engine утечёт.
  **Исправление:** обернуть блок в `try/finally`:
  ```python
  engine = create_async_engine(settings.DATABASE_URL)
  try:
      result = await sync_catalog(_sf(engine, expire_on_commit=False))
  finally:
      await engine.dispose()
  ```
  Альтернатива: передавать `session_factory` из глобального контекста (например, через middleware data или bot/dispatcher).

### bot/handlers/lead.py

- **[критичность: низкая] Корректность: состояние FSM `LeadForm.time` можно пройти любым текстом без пустой строки.**
  Строка 217: `if message.text != "Пропустить"` — обрабатывает поле как skip только при точном совпадении строки. Это корректно само по себе, но кнопка «Пропустить» в `step_keyboard` является `ReplyKeyboardMarkup`, а не callback, и строковое сравнение зависит от локализации клиента. Риск минимален, но хрупко.

- **[критичность: низкая] Читаемость: `start_contact` использует `event` без аннотации типа.**
  Строка 119: `async def start_contact(event, state, session)` — параметр `event` не аннотирован. aiogram 3.x поддерживает `Union[CallbackQuery, Message]` как тип; без аннотации IDE и mypy не могут проверить использование `event.from_user`, `event.answer()` и т.д.

### bot/middleware.py

- **[критичность: средняя] Производительность + безопасность: `CallbackDebounceMiddleware._seen` неограниченно растёт под нагрузкой.**
  Словарь `_seen` чистится при каждом входящем callback: перебираются все ключи, старше `window_seconds`. При тысячах пользователей и коротком окне (1.2 с) в памяти накапливаются записи от всех предыдущих сессий до момента следующего callback. Это не критично для небольшого бота, но стоит заменить на `collections.OrderedDict` с ограничением или TTL-кеш.

- **[критичность: низкая] Корректность: `DbSessionMiddleware` не делает rollback при исключении.**
  Сессия открывается через `async with self.session_factory() as session`. `async_sessionmaker` по умолчанию делает rollback при выходе из контекста с исключением — это правильно. Замечание снято.

### bot/services/lead.py

- **[критичность: низкая] Корректность: `normalize_phone` возвращает одинаковый результат в обеих ветках последней строки.**
  Строка 25:
  ```python
  return f"+{digits}" if not raw.strip().startswith("+") else f"+{digits}"
  ```
  Обе ветки тернарного выражения возвращают `f"+{digits}"`. Условие бессмысленно и вводит в заблуждение. Это не баг (результат всегда правильный), но явная логическая ошибка.
  **Исправление:** `return f"+{digits}"`.

### bot/services/admin.py / bot/repositories/admin.py

- **[критичность: низкая] Читаемость: пользовательские данные в карточке заявки не экранируются для HTML.**
  `format_lead_card` и `format_admin_notification` вставляют `lead.name`, `lead.phone`, `lead.city` напрямую в строку, которая затем рендерится с `parse_mode="HTML"`. Имена вида `<Иван>` или `AT&T` могут нарушить разметку или вызвать `TelegramBadRequest`.
  **Исправление:** обернуть пользовательские поля в `html.escape()`, как это уже сделано в `format_confirmation`.

### bot/parser/parser.py

- **[критичность: средняя] Безопасность: `ssl._create_unverified_context()` отключает проверку TLS-сертификата.**
  Строка 86: `context=ssl._create_unverified_context()`. Обращение к `altairika.ru` происходит без верификации сертификата, что делает соединение уязвимым к MITM-атаке. Функция `_create_unverified_context` также является внутренней (`_`-префикс) и может измениться.
  **Исправление:** убрать параметр `context` (использовать дефолтный проверяющий контекст) или передать `ssl.create_default_context()`.

- **[критичность: низкая] Производительность: список `items` переопределяется дважды.**
  Строки 496 и 503: `items = list(products_by_title.values())` вызывается дважды. Первый вызов перезаписывается `await asyncio.gather(...)` (который enriches items in-place), затем второй вызов нужен для финального результата. Первое присваивание (строка 496) используется только как источник для `asyncio.gather`, но после него `products_by_title.values()` содержит те же объекты. Код корректен, но вводит в заблуждение.

### docker-compose.yml

- **[критичность: высокая] Безопасность: учётные данные PostgreSQL захардкожены в файле репозитория.**
  `POSTGRES_PASSWORD: secret` и аналогичные строки вшиты в `docker-compose.yml`. Этот файл, по всей видимости, коммитится в git. Секреты должны подставляться через `.env` (уже используется для `BOT_TOKEN`) или внешний секрет-менеджер.
  **Исправление:** заменить на переменные окружения:
  ```yaml
  environment:
    POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
  ```
  и добавить `POSTGRES_PASSWORD` в `.env` (который в `.gitignore`).
  Важно: `DATABASE_URL` в секции `bot` также содержит `secret` в plain text прямо в compose-файле (строка 25), что отдельно нарушает то же правило.

- **[критичность: низкая] Отсутствует `user:` в сервисе `bot`** — контейнер запускается от `root`. Для production рекомендуется добавить `user: "1000:1000"` или создать отдельного пользователя в Dockerfile.

### alembic/versions/

- **[критичность: низкая] Миграция 0001: отсутствует `onupdate` для `updated_at` в `catalog_items` и `leads`.**
  В миграции колонки `updated_at` создаются только с `server_default`, без триггера обновления. В SQLAlchemy-модели указан `onupdate=func.now()`, но это работает только через ORM (SQLAlchemy генерирует запрос с явным значением). При прямом UPDATE через SQL (миграция, внешний инструмент) поле не обновится автоматически. Если это требование — нужен `ON UPDATE` триггер или `server_onupdate` в миграции.

## Что сделано хорошо

- **Архитектура слоёв** выдержана чисто: handlers → services → repositories. Бизнес-логика не просачивается в handlers.
- **FSM для форм** реализован грамотно: защита от двойного сабмита (`submitting` флаг), корректное восстановление состояния при прерывании, отдельный шаг подтверждения перед записью.
- **`CallbackDebounceMiddleware`** — нетривиальная, но полезная защита от «двойного нажатия», реализована без блокировок.
- **`show_text_screen` / `show_photo_screen`** — правильный паттерн: сначала `edit_message`, при ошибке `answer` + `delete`. Соответствует UX-требованию «навигация через редактирование».
- **Парсер** хорошо изолирован, graceful degradation при недоступности сайта, семафор на параллельные запросы к страницам (8 штук), адекватное логирование.
- **`log_event`** не прерывает основной флоу при ошибке записи — правильное решение.
- **Миграции** версионированы последовательно, `downgrade()` полностью реализован.
- **Экранирование HTML** (`html.escape`) применено в `format_item_text` и `format_confirmation` — потенциальные XSS-точки закрыты.
- **Проверка администратора** реализована единообразно через `_is_admin()` на каждом handler.
