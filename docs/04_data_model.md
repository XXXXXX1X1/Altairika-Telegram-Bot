# 04 — Модель данных

## 1. Основные сущности

### 1.1 Позиция каталога (`CatalogItem`)

| Поле | Тип | Описание | Источник |
|------|-----|----------|---------|
| `id` | string / int | Уникальный идентификатор | Генерируется при парсинге |
| `title` | string | Название | Сайт |
| `description` | text | Полное описание | Сайт |
| `short_description` | text | 1–2 строки для превью | Генерируется из description |
| `category` | string | Категория отображения в боте | Парсер / импорт |
| `tags` | list[string] | Дополнительные теги | Сайт (опционально) |
| `image_url` | string \| null | URL изображения / постера | Сайт (если есть) |
| `price` | string \| null | Цена в рублях | Сайт (если есть) |
| `duration` | string \| null | Длительность | Сайт (если есть) |
| `age_rating` | string \| null | Возрастной рейтинг (`0+`, `5+`, `7+`, `12+`, `16+`, `18+`) | Сайт → нормализация парсером |
| `url` | string | Ссылка на страницу сайта | Сайт |
| `is_active` | bool | Отображать ли в боте | Управляется вручную или по сайту |
| `updated_at` | datetime | Дата последнего обновления | Система |

> [Требует данных] Конкретный набор полей уточняется после анализа реального каталога на сайте. Часть полей может отсутствовать — это нормально, бот должен корректно отображать карточку с неполными данными.

> [Решение] Поле `short_description` генерируется автоматически при парсинге (первые N символов description или отдельное поле, если сайт его предоставляет). Хранится отдельно, чтобы не усекать каждый раз при показе.

### 1.2 Категория (`Category`)

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | string | Идентификатор |
| `name` | string | Название категории |
| `order` | int | Порядок отображения |
| `item_count` | int | Количество позиций (денормализовано для скорости) |

> [Решение] Категории формируются автоматически при синхронизации каталога.

> [Текущий статус] Автоматический парсер загружает канонический каталог сайта как одну категорию `Каталог`. Это гарантирует, что в БД попадают все 166 фильмов. Более детальная рубрикация может быть добавлена отдельным этапом.

## 2. Источники данных

### 2.1 Каталог

| Источник | Метод | Статус |
|----------|-------|--------|
| altairika.ru | Tilda API через страницу `catalog_full` | Используется |
| Ручной ввод | Форма администратора или прямое редактирование БД | Резервный |

> [Решение] Парсер читает HTML страницы `catalog_full`, извлекает идентификаторы блоков Tilda Store и получает сами карточки через `https://store.tildaapi.com/api/getproductslist/`.

> [Решение] Если найдено несколько блоков каталога, выбирается блок с максимальным `total`. На текущем сайте это каталог на 166 фильмов.

> [Решение] Парсинг запускается по расписанию раз в день. Изменения в каталоге синхронизируются автоматически. Новые позиции добавляются, изменённые обновляются, удалённые помечаются `is_active = false`.

### 2.2 Данные о конкурентах

| Источник | Метод |
|----------|-------|
| Сайты конкурентов | Ручной мониторинг |
| Открытые рейтинги (2ГИС, Яндекс и др.) | Частичный парсинг |
| Отраслевые агрегаторы | Ручной мониторинг |

> [Гипотеза] Конкуренты определяются при первичном исследовании. Данные о них обновляются вручную (1–2 раза в квартал достаточно для старта).

## 3. Требования к данным каталога

### Обязательные поля (без них карточка не отображается)

- `title`
- `description`

### Желательные поля (влияют на качество карточки)

- `category` — нужна для фильтрации; сейчас по умолчанию используется общая категория `Каталог`
- `image_url` — визуальная привлекательность карточки
- `price` — часто задаваемый вопрос

### Опциональные поля

- `duration`, `age_rating`, `tags`

> [Решение] Бот корректно отображает карточку даже с минимальным набором полей. Пустые поля просто не показываются — нет «Цена: не указана».

---

## 4. Хранение данных

> [Решение] Основная база данных — **PostgreSQL**. Выбор обоснован:
> - Реляционная структура данных подходит лучше, чем документная (MongoDB).
> - Хорошая поддержка в Python-экосистеме.
> - Достаточно для нагрузок этого проекта.
> - Простое резервное копирование.

> [Гипотеза] На старте можно использовать SQLite для быстрого прототипирования, затем перейти на PostgreSQL при деплое. Решение зависит от выбранного хостинга.

---

## 5. Требования к персональным данным

- Перед сохранением контакта пользователь явно подтверждает согласие на обработку персональных данных (текст + ссылка на политику).
- Номер телефона хранится в базе в открытом виде (зашифровывать — [Улучшение] на второй фазе).
- Данные не передаются третьим лицам, кроме оператора.
- Пользователь может запросить удаление данных через команду `/delete` или обращение к оператору.

> [Требует данных] Политика конфиденциальности и точный текст согласия — предоставляется заказчиком или юристом.
