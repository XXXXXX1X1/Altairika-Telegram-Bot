# 09. AI-ассистент — реализация


> **Статус:** реализовано (Фаза 8)  
> Дата реализации: апрель 2026 (~5 часов разработки)

---

## Обзор

AI-ассистент встроен в `freetext_router` — перехватчик свободного текста вне FSM-форм. Если `OPENROUTER_API_KEY` не задан, бот возвращает заглушку с кнопками FAQ/оператор. При заданном ключе — запускается полный pipeline.

**Две модели через OpenRouter API:**
- `AI_ROUTING_MODEL = openai/gpt-4.1-nano` — все служебные вызовы: routing, JSON-ответы, извлечение параметров, ранжирование. Быстрая и дешёвая.
- `AI_MODEL = openai/gpt-4o-mini` — финальный ответ пользователю. Более качественная.

Обе модели меняются через `.env` без изменения кода.

## Полный pipeline — шаг за шагом

### 1. Детектирование UI-команды (без LLM)

Перед любым обращением к модели `freetext.py` проверяет, не является ли текст навигационной командой:

```
«открой каталог» / «открыть каталог» / «покажи каталог» → открывает каталог
«открой франшизу» / «покажи франшизу» → открывает раздел франшизы
«главное меню» / «в меню» → главное меню
```

Если контекст диалога говорит о каталоге/франшизе, однословные «да», «давай», «покажи» тоже перехватываются. Это экономит токены на банальных навигационных запросах.

### 2. Загрузка сессии из БД

```python
ai_state = await load_state(session, message.from_user.id)
```

Возвращает словарь с историей диалога, активным intent и параметрами предыдущего подбора. Если сессии нет или она истекла — пустой dict.

### 3. Определение intent — двухуровневая система

**Уровень 1: LLM-маршрутизатор** (`ai_decision.py`)

Первый вызов модели (`AI_ROUTING_MODEL`). Модель получает:
- Системный промпт с правилами routing
- Текущее состояние сессии (active_intent, параметры, последние 6 сообщений)
- Текст пользователя

Возвращает JSON:
```json
{
  "intent": "movie_selection",
  "action": "run_search",
  "use_current_movie": false,
  "open_current_movie_card": false,
  "continue_current_flow": true,
  "confidence": 0.92,
  "reason": "user specified grade and theme"
}
```

Если `confidence < 0.45` или LLM вернул некорректный ответ — переходим на уровень 2.

**Уровень 2: Эвристический скоринг** (`ai_router.py`)

Без вызова LLM. Каждое ключевое слово из запроса добавляет очки нужному intent. Примеры:

```
«записаться» / «забронировать» / «перезвоните»  → +4 балла lead_booking
«франшиза» / «паушальный» / «роялти»             → +4 балла franchise_info
«3 класс» / «для детей» / «до 15 минут»          → +3 балла movie_selection
«конкурент» / «vizerra» / «vr concept»           → +4 балла competitor_compare
```

Побеждает intent с наибольшим счётом. Если у нескольких equal — приоритет: lead > compare > franchise > movie > faq > company > general_chat.

**Уровень 3: Контекстная ветвь** (`ai_branch.py`)

Дополнительная эвристика поверх детектированного intent. Если пользователь уже был в каком-то сценарии, короткие реплики остаются в нём:

```
Active: franchise_info + «да», «расскажи», «интересно»  → остаёмся в franchise_info
Active: competitor_compare + «цена», «контент», «франшиза» → остаёмся в competitor_compare
Active: movie_selection + есть ai_current_item_id + «расскажи подробнее» → movie_details (текущий фильм)
```

Это позволяет вести многошаговый диалог без лишних вызовов модели.

### 4. Маршрутизация по intent

```
lead_booking / lead_franchise → ответ «нажмите кнопку» + кнопка формы (без LLM)
movie_selection              → run_ai_pick_flow() (FSM ai_movie.py)
open_current_movie_card      → send_movie_card_message() (без LLM)
все остальные                → generate_answer()
```

### 5. Генерация ответа (`ai_answer.py`)

**Шаг 5.1: Подготовка данных каталога**

Если `movie_selection`:
- `extract_movie_params()` — AI-парсинг параметров (тема, класс, возраст, длительность) через `AI_ROUTING_MODEL`, fallback на regex
- `find_relevant_films()` — поиск с учётом параметров (см. подробнее ниже)

Если `movie_details`:
- `extract_movie_title_candidate()` — regex-извлечение названия из текста
- `find_movie_by_title()` — точный поиск по названию через SequenceMatcher
- При промахе → `find_similar_movies()` → шаблонный ответ со списком похожих (без LLM-вызова)

**Шаг 5.2: Сборка контекста** (`ai_context.py`)

Для каждого intent — своя комбинация источников. Контекст намеренно минимальный, чтобы не засорять prompt и не платить лишние токены:

```
movie_selection  → company_knowledge + найденные фильмы (3-5 шт)
franchise_info   → company_knowledge + FAQ + franchise_content (4 секции)
competitor_compare → company_knowledge + таблица сравнения (без FAQ)
faq_answer       → company_knowledge + FAQ (без franchise, без compare)
company_info     → company_knowledge + FAQ
```

`company_knowledge` (`docs/work/10_company_knowledge.md`) кешируется в памяти при первом обращении — файловый I/O только один раз за жизнь процесса.

**Шаг 5.3: Системный промпт**

Строится из двух частей:

```
[Базовые правила] — общие ограничения: отвечай только по контексту, не выдумывай,
                    кратко, только русский, не ссылайся на источник явно
       +
[Инструкция по intent] — конкретика для текущего сценария:
  movie_selection:   «выбери 2-3 подходящих, объясни почему, задай один вопрос если мало данных»
  franchise_info:    «по данным franchise_content, в конце CTA на заявку»
  competitor_compare: «из таблицы сравнения, без звёздочек-списков, не придумывай цифры»
  ...
```

**Шаг 5.4: Вызов модели**

```python
answer = await call_llm(system_prompt, user_text, history=history)
# history — последние 8 сообщений из ai_sessions (4 полных обмена)
```

Модель видит: правила + данные + короткую историю + текущий вопрос. Не всю БД, не весь каталог — только то, что нужно для ответа.

**Шаг 5.5: Сохранение состояния**

```python
await update_state(db, user_id, intent, updated_state)
```

В `ai_sessions` сохраняется: обновлённая история + параметры подбора + id/title текущего фильма + active_intent.

### 6. Ответ пользователю

Текст ответа + `after_ai_keyboard(intent)` — набор кнопок, релевантных текущему сценарию.

## Алгоритм подбора фильмов (детально)

Это самый сложный сценарий — разберём полностью.

### Шаг 1: Извлечение параметров

**AI-парсинг** (`ai_movie_params.py`):

Первый LLM-вызов (`AI_ROUTING_MODEL`, max_tokens=180). Запрос пользователя + история + текущие параметры → JSON:

```json
{
  "theme": "космос",
  "grade": 3,
  "age": null,
  "audience": null,
  "duration": "d15",
  "needs_clarification": false,
  "clarification_reason": ""
}
```

**Regex-fallback** (параллельно): если LLM-вызов упал или вернул мусор, regex-паттерны извлекают класс/возраст/длительность/тему из текста. Результаты объединяются: LLM имеет приоритет, regex заполняет пропуски.

**Параметры:**

| Параметр | Значения | Как определяется |
|---------|---------|-----------------|
| `theme` | «космос», «природа», ... | LLM или keyword-маппинг |
| `grade` | 1–11 | `(\d{1,2})\s*класс` |
| `age` | число | `(\d{1,2})\s*лет` |
| `audience` | preschool / primary / secondary | «детский сад», «начальная школа», ... |
| `duration` | d5 / d15 / d30 / d30p | `до N минут`, `N минут` |

**Маппинг класса → age_rating:**
```
1–3 класс → 6+, 7+
4 класс   → 7+, 10+
5–6 класс → 10+, 12+
7 класс   → 12+
8–9 класс → 12+, 16+
10–11 класс → 16+
```

### Шаг 2: SQL-фильтрация

```sql
SELECT * FROM catalog_items
WHERE is_active = true
  AND (age_rating IN ('7+', '10+') OR age_rating IS NULL)   -- если grade/age задан
  AND (duration IN ('d5', 'd15') OR duration IS NULL)        -- если duration задан
```

Если под фильтр ничего не попало — повтор без возрастных фильтров (только по теме).

### Шаг 3: Скоринг по теме

Каждый фильм получает балл:
- +3 если тема из запроса совпадает с тегами фильма (keyword-маппинг)
- +1 если тема встречается в description
- +1..4 за пересечение токенов raw_query с title/description/tags

### Шаг 4: AI-ранжирование (при необходимости)

Если тема не попала в keyword-маппинг или score у всех фильмов = 0:

1. Семантический shortlist: берём top-12 кандидатов по токенному пересечению
2. Второй LLM-вызов (`AI_ROUTING_MODEL`, max_tokens=220):
   ```
   Запрос пользователя + список кандидатов (id + title + description + tags)
   → { "matched_ids": [47, 12, 88] }
   ```
3. Возвращаем только фильмы из matched_ids в исходном порядке

Этот шаг срабатывает только когда keyword-алгоритм не справился — например «что-нибудь для выпускного» или «про морских животных».

### Шаг 5: Финальный ответ

Найденные 3-5 фильмов передаются в `generate_answer()` как часть контекста. Модель (`AI_MODEL`) формулирует человеческий ответ с объяснением почему эти фильмы подходят.

## FSM-подбор через кнопку «Подобрать фильм»

Отдельный сценарий через inline-кнопку главного меню. Пользователь попадает в состояние `AiPick.waiting` и видит вопрос с примерами запросов.

```
[кнопка] → AiPick.waiting → текст пользователя
                                    │
                     ┌──────────────┼──────────────────┐
                     ▼              ▼                  ▼
               show_themes     ask_clarification    run_search
             (список тем)    (уточняющий вопрос) (показ карточек)
                                                       │
                                           ┌───────────┴──────────┐
                                     [← →] листание           [Уточнить]
                                     карточек               AiPick.refine
                                                                   │
                                                          уточнение параметров
                                                          новый поиск
```

**Управление:**
- `aip:nav idx=N` — показать N-ю карточку (edit в том же сообщении)
- `aip:newtopic` — сбросить параметры, начать заново (с сохранением снапшота для «назад»)
- `aip:refine` — сохранить текущие параметры, добавить уточнение
- `aip:back` — восстановить предыдущую подборку из снапшота
- `aip:exit` — очистить FSM, главное меню

При переключении на другой intent внутри FSM (например «хочу записаться») — FSM очищается, пользователь получает релевантный ответ.

## Сессионная память

**Таблица:** `ai_sessions` (миграция `0003_ai_sessions.py`)

| Поле | Тип | Описание |
|------|-----|---------|
| `telegram_user_id` | bigint PK | Пользователь |
| `active_intent` | varchar(100) | Текущий сценарий |
| `state_json` | text | JSON с историей и параметрами |
| `updated_at` | timestamptz | Обновляется при каждом обмене |
| `expires_at` | timestamptz | TTL (не используется для auto-clean, только для справки) |

**Структура `state_json`:**

```json
{
  "_active_intent": "movie_selection",
  "params": {
    "theme": "космос",
    "grade": 3,
    "duration": "d15",
    "raw_query": "космос для 3 класса до 15 минут",
    "last_recommended_ids": [12, 47, 88]
  },
  "ai_current_item_id": 47,
  "ai_current_item_title": "Космическое путешествие",
  "ai_flow_step": "showing_results",
  "history": [
    {"role": "user", "content": "фильм про космос для 3 класса"},
    {"role": "assistant", "content": "Вот 3 варианта..."},
    {"role": "user", "content": "а покороче?"},
    {"role": "assistant", "content": "Хорошо, вот до 15 минут..."}
  ]
}
```

**Лимит истории:** 8 сообщений (4 обмена). Старые обрезаются.  
**TTL:** 30 минут (через `AI_SESSION_TTL_MINUTES`). Очистка не автоматическая — при следующем запросе система работает с устаревшим контекстом, но это не критично: params могут быть нерелевантны, но не ломают логику.

## Пример полного трейса диалога

**Пользователь:** «что есть про природу для начальной школы?»

```
1. _detect_ui_action() → null (не навигация)
2. load_state() → {} (нет сессии)
3. analyze_dialog_scenario() →
   { intent: "movie_selection", action: "run_search", confidence: 0.88 }
4. detect_intent() (fallback не нужен, confidence > 0.45)
5. intent = "movie_selection" → run_ai_pick_flow()
6. extract_movie_params("что есть про природу для начальной школы") →
   { theme: "природа", audience: "primary", raw_query: "природа начальная школа" }
7. find_relevant_films(params) →
   SQL: age_rating IN ('6+', '7+', '10+')
   Скоринг: фильмы с тегом "природа" получают +3
   Результат: ["Живая природа", "Мир животных", "Экосистемы"]
8. Показ первой карточки "Живая природа", кнопки ← → Уточнить Другая тема
9. update_state() → сохранено в ai_sessions
```

**Пользователь:** «а покороче?»

```
1. load_state() → { theme: "природа", audience: "primary", ... history: [...] }
2. analyze_dialog_scenario() →
   { intent: "movie_selection", action: "run_search",
     continue_current_flow: true, confidence: 0.79 }
3. extract_movie_params("а покороче?", existing_state) →
   AI видит контекст, добавляет duration: "d15" к существующим параметрам
4. find_relevant_films({ theme: "природа", audience: "primary", duration: "d15" }) →
   SQL: age_rating IN ('6+', '7+', '10+') AND duration в коротких
   Результат: ["Мир животных (12 мин)"]
5. Показ новой карточки с обновлённым заголовком «Нашёл 1 фильм — тема природа, до 15 мин»
```

## Количество LLM-вызовов на запрос

| Ситуация | Вызовы | Модели |
|---------|--------|-------|
| Навигационная команда | 0 | — |
| lead_booking / lead_franchise | 1 (routing) | ROUTING_MODEL |
| Обычный вопрос (company, faq, franchise, compare) | 2 (routing + ответ) | ROUTING_MODEL + AI_MODEL |
| movie_selection (тема в keyword-маппинге) | 3 (routing + params + ответ) | ROUTING_MODEL×2 + AI_MODEL |
| movie_selection (тема вне маппинга) | 4 (routing + params + ranking + ответ) | ROUTING_MODEL×3 + AI_MODEL |
| movie_details (фильм найден точно) | 2 (routing + ответ) | ROUTING_MODEL + AI_MODEL |
| movie_details (фильм не найден) | 1 (routing) | ROUTING_MODEL (шаблонный ответ) |

## Fallback-поведение

| Ситуация | Поведение |
|---------|---------|
| `OPENROUTER_API_KEY` не задан | Статическая заглушка + кнопки FAQ / оператор |
| LLM timeout (20 сек) | None → fallback текст + fallback кнопки |
| LLM вернул некорректный JSON | Переход на эвристики (ai_router) |
| LLM confidence < 0.45 | Переход на эвристики (ai_router) |
| `general_chat` + LLM упал | Статический приветственный текст |
| Фильм не найден (есть похожие) | Список похожих, без LLM |
| Фильм не найден вообще | Предложение уточнить или написать параметры |

## Ограничения

- Нет векторного поиска (pgvector / RAG) — только keyword/token matching + LLM-ранжирование
- История диалога ограничена 8 сообщениями
- Сессия AI переживает перезапуск бота (PostgreSQL), FSM-состояние подбора — нет (MemoryStorage)
- База знаний кешируется in-memory — для обновления нужен перезапуск бота
- Auto-expiry сессий не реализован — TTL только информационный
- Один администратор на инстанс
