# Как разрабатывался проект с Claude Code

> Этот документ о процессе разработки — как велась работа, что использовалось и в каком порядке.

---

## Затраченное время

| Этап | Время |
|------|-------|
| Составление документации (`docs/01–06`) | 1 час |
| Реализация (все фазы кода) | 6 часов |
| Тестирование и доработка интерфейса | 3 часа |
| **Итого** | **10 часов** |

> Реализация заняла больше всего времени в том числе из-за лимитов Claude Code — приходилось ждать сброса лимитов и разбивать работу на отдельные сессии.

## С чего начали — документация

Прежде чем писать код, была создана полная документация проекта в папке `docs/`:

- `01_product_brief.md` — что строим и для кого
- `02_user_scenarios.md` — все сценарии использования
- `03_ux_design.md` — каждый экран, тон бота, правила навигации
- `04_data_model.md` — структура БД
- `05_architecture.md` — стек и модули
- `06_roadmap.md` — порядок реализации по фазам

В `CLAUDE.md` прописали правило: перед любой работой с кодом читать `docs/` по порядку. Это давало Claude контекст и не позволяло принимать произвольные решения на ходу.

## Настройка поведения через правила

В `.claude/rules/` лежат три файла, которые загружаются автоматически в каждой сессии.

**`план-выполнение-проверка`** — любая задача проходит три шага: сначала разобраться и наметить план, потом реализовать, потом обязательно проверить. Без проверки задача не считается выполненной. Это реально снизило количество итераций — меньше «ой, не то сделал».

**`правила-поведения`** — Claude задаёт уточняющие вопросы перед крупными задачами, не додумывает за тебя, после каждой задачи перечисляет что конкретно проверил. Если задача важная — запускает независимый субагент на проверку.

**`стиль-общения`** — неформально, по делу, без соглашательства. Если что-то сделано неоптимально — скажет прямо.

## Разработка по фазам

Код писался итеративно по фазам из роадмапа:

1. Инфраструктура — PostgreSQL, Docker, Alembic, polling
2. Каталог — парсер altairika.ru, карточки, фильтры
3. Лиды — FSM-формы, подтверждение, уведомления администратору
4. FAQ, франшиза, сравнение с конкурентами
5. Аналитика, панель администратора

Каждую фазу закрывали коммитом через скилл `/commit`.

---

## Скилл `/commit`

Вместо того чтобы каждый раз вручную писать git-команды, использовался скилл `/commit`. Он сам добавляет нужные файлы (без `.env`, без `tmp/`), делает коммит с сообщением на русском и пушит.

**Один раз это сломалось**: добавили `POSTGRES_*` переменные в `.env` и сразу закоммитили — но не проверили что бот запускается. При следующем старте вылетел `pydantic ValidationError: Extra inputs are not permitted` — модель `Settings` не знала про эти переменные. Пришлось добавить `"extra": "ignore"` в `model_config`. После этого зафиксировали в памяти: **коммитить только после проверки запуска**.

## Code Review через субагента

После завершения основной разработки запустили субагента `code-reviewer`. Он читает код без контекста сессии — смотрит свежим взглядом. Результат сохранили в `docs/07_code_review.md`.

Нашёл реальные баги:

- `faq.py`: `question = await save_user_question(...)` — результат не присваивался переменной, а строкой ниже использовался `question.id`. Упало бы с `NameError` при первом же вопросе пользователя.
- `docker-compose.yml`: пароль БД `secret` захардкожен прямо в файле репозитория.
- `start.py`: при ошибке в `/sync` engine не закрывался — не было `try/finally`.
- `parser.py`: использовался `ssl._create_unverified_context()` — соединение без проверки сертификата.
- `services/lead.py`: `return f"+{digits}" if not raw.strip().startswith("+") else f"+{digits}"` — обе ветки возвращают одно и то же.

Всё исправили до деплоя.

## Параллельный аудит текстов

Для проверки пользовательских текстов запустили **два субагента одновременно** в фоне:

**`reviewer`** — проверял все строки которые видит пользователь: орфографию, пунктуацию, логику, HTML-теги. Нашёл:
- Мёртвый код в FAQ — `if text == "Отмена"`, которая никогда не срабатывает, потому что кнопка отмены инлайновая
- `"Скрываю клавиатуру…"` — технический текст иногда мелькал у пользователя
- `🆕 ФРАНШИЗА` капслоком vs `🆕 Новая заявка` в обычном регистре — непоследовательно
- `"еще"` без мягкого знака
- `"Altairika"` и `"Альтаирика"` вперемешку в одном разделе

**`research`** — сверял код с документацией, искал расхождения между спецификацией и реализацией. Нашёл:
- Раздел «Рынок и конкуренты» полностью реализован (`compare.py`, обработчики), но кнопка в меню франшизы не добавлена — раздел недоступен пользователю
- Кнопка «Задать вопрос» в карточке каталога ведёт в общий FAQ вместо привязки к конкретной позиции
- Финальное сообщение для заявки на франшизу не говорит «в течение 1 рабочего дня» как описано в доках

Пока агенты работали параллельно — не нужно было ждать первого чтобы запустить второго.

## Отладка Docker на macOS

При сборке образа с нуля `pip install` падал с ошибкой несовпадения хешей. Оказалось — SSL EOF на macOS M-чип: пакеты скачивались как пустые файлы. Исправлено добавлением флагов в `Dockerfile`:

```dockerfile
RUN pip install --no-cache-dir \
    --trusted-host pypi.org \
    --trusted-host files.pythonhosted.org \
    -r requirements.txt
```

После инцидента с коммитом до проверки записали в память:
